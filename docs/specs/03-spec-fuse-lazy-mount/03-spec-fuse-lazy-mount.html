03-spec-fuse-lazy-mount

Sequence 03 · Feature: fuse-lazy-mount · Phase 1

Introduction/Overview

This feature pivots Plan 015 to match the team's macOS-first development reality. The primary product plan is to make the existing devspace mount prototype understandable, diagnosable, and testable on developer Macs with macFUSE; Linux hosted CI remains a follow-up automation safety net for repeatable regression coverage.

Goals

  • Produce observed GitHub Actions evidence for Linux FUSE GO or NO-GO status.
  • Make macOS the first-class user path for FUSE lazy mount readiness, setup, and troubleshooting.
  • Keep normal local and CI verification FUSE-free.
  • Use Linux hosted CI as opt-in automated coverage after macOS product guidance is explicit.
  • Update plan, design, proof, and validation artifacts so Plan 015 is no longer blocked.

User Stories

  • As a macOS DevSpace developer, I want to understand whether my Mac can run the lazy mount prototype so that I can try the feature without guessing at FUSE setup.
  • As a DevSpace maintainer, I want to document macOS readiness before CI automation so that the plan follows the team's real development platform.
  • As a CLI contributor, I want to run FUSE tests only on capable Linux hosts so that normal development remains portable.
  • As a reviewer, I want to see macOS readiness evidence and Linux CI evidence separately so that product fit and automation fit are not conflated.

Demoable Units of Work

Each unit is a thin, verifiable slice of the macOS-first Plan 015 decision path.

Unit 1: macOS Local FUSE Readiness

Purpose: Put the developer Mac path first by documenting requirements, diagnostics, and the safe fallback when macFUSE is unavailable.

Functional Requirements:

  • The system shall document macOS as the primary developer target for the lazy mount prototype.
  • The system shall record local macOS readiness evidence without installing macFUSE or changing system security settings automatically.
  • The user shall have explicit setup, verification, fallback, and next-proof commands for macFUSE-based local testing.

Proof Artifacts:

  • CLI output from sw_vers and macFUSE presence checks demonstrates the current local readiness state.
  • Documentation diff demonstrates macOS-first setup and fallback guidance.

Unit 2: Linux Hosted CI Safety Net

Purpose: Preserve the successful Linux probe as automated regression coverage, but position it as a follow-up safety net rather than the product-first plan.

Functional Requirements:

  • The system shall capture the Linux probe run URL, runner image details, and pass output in proof artifacts.
  • The system shall add a Linux-only mount-integration job after hosted Linux FUSE is proven GO.
  • The system shall keep make verify and the default CI verify job FUSE-free.

Proof Artifacts:

  • GitHub Actions run URL and log excerpt demonstrate observed hosted Linux FUSE behavior.
  • CI workflow diff demonstrates the opt-in Linux integration job.

Unit 3: Plan Reconciliation and Follow-Up Boundaries

Purpose: Update Plan 015 and SDD artifacts so the final direction is macOS-first with Linux automation follow-up.

Functional Requirements:

  • The system shall update docs/architecture/fuse-lazy-mount.md with separate macOS local readiness and Linux CI feasibility sections.
  • The system shall update plans/README.md with a final Plan 015 status and one-line rationale.
  • The system shall identify hosted macOS CI, macOS 26 FSKit exploration, richer status cards, and path-vs-name decisions as follow-up work.

Proof Artifacts:

  • Documentation diff demonstrates the macOS-first pivot and Plan 015 reconciliation.
  • make verify output demonstrates the normal gate remains FUSE-free.

Non-Goals (Out of Scope)

  1. Automatic macFUSE installation: This workflow will not install macFUSE or alter macOS security settings automatically.
  2. Hosted macOS FUSE CI: Hosted macOS runner support remains a future evaluation because macFUSE approval and kernel/system extension loading are not reliable assumptions on ephemeral hosted runners.
  3. Product behavior changes: The existing devspace mount behavior should not change except for testability fixes required to validate documented behavior.
  4. Richer status cards: dirty repo, missing .env, and setup hint display remain follow-up work.
  5. Path-vs-name mount design: Long-term virtual directory decisions remain out of scope.

Design Considerations

No specific UI design requirements identified. HTML artifacts should be reviewer-friendly and self-contained.

Repository Standards

  • Follow Go formatting and keep tests beside implementation in internal/devspace/.
  • Use make verify as the normal local CI gate.
  • Keep generated build outputs out of source and avoid committing secrets or workspace state.
  • Use Conventional Commit style for implementation commits.
  • Preserve the repository instruction that default CLI flows must work without FUSE.

Technical Considerations

The existing prototype uses github.com/hanwen/go-fuse/v2/fs. macOS support depends on macFUSE or a compatible implementation installed and approved by the user. GitHub-hosted runner software changes over time, so Linux automation must rely on observed workflow evidence rather than local inference. Build tags such as linux && fusetest keep FUSE tests opt-in and portable.

Security Considerations

  • Proof artifacts must not include GitHub tokens, hosted tokens, age identities, or real environment values.
  • The implementation must not install macFUSE, request elevated macOS permissions, or change system extension settings automatically.
  • The probe workflow must not require repository secrets.
  • FUSE tests must use temporary directories and local bare Git remotes only.

Success Metrics

  1. macOS readiness: local macOS state is documented with next-proof commands and fallback behavior.
  2. Linux automation: Linux hosted runner status is documented as GO or NO-GO with an observed run.
  3. Default verification: make verify exits 0 without FUSE.
  4. Plan reconciliation: Plan 015 is no longer BLOCKED without evidence.
  5. Review gate: coderabbit review --agent is run after implementation and still-valid findings are addressed.

Open Questions

No blocking open questions. Assumption: the user authorized temporary branch push and GitHub Actions execution by requesting implementation of this plan.
