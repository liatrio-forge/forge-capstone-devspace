1. Command Name

What should the dashboard subcommand be called?

Current best-practice context: Charm-family CLIs generally keep the root command non-interactive (safe for scripts) and expose the TUI behind a short subcommand; gh's ecosystem popularized dash as a separate extension name, while k9s-style tools own the whole binary. Option (D) would change the root command's scripted behavior and contradicts the spec-04 posture of script-safe defaults.
Recommended answer(s): (A)

Why these are recommended:

  • (A) ui is the shortest name that stays accurate if the dashboard later gains non-dashboard panes (logs, conflicts from spec 06), whereas (C) narrows the name to one layout.
  • (B) tui is developer-jargon accurate but reads worse in docs ("run devspace tui"); (A) reads naturally.
  • (D) is rejected-by-default: it would break every script that runs bare devspace expecting help/exit-2 behavior, violating the script-compat posture locked in spec 04.
✅ DECISION (Jacob): (A) devspace ui. Non-breaking (bare devspace keeps help/exit-2), reads naturally in docs, and stays accurate if the dashboard later gains non-dashboard panes (e.g. spec 06 conflict view). Endorses the recommendation as-is.

2. Live Watch Pane Mechanism

The decided shape includes "live watch events." How should the dashboard's live refresh work under the hood?

Current best-practice context: The repo already has a debounced, ignore-rule-aware fsnotify pipeline (watch.go: WatchWorkspace + OnRefresh callback + scoped refresh) whose per-refresh withAppLock usage is the established cross-process-safety pattern; reusing it means the dashboard inherits debouncing, ignore rules, and lock discipline for free. A polling loop would duplicate that machinery.
Recommended answer(s): (D)

Why these are recommended:

  • (D) reuses the proven watcher exactly as (A) does, but the manual-refresh key and --no-watch escape hatch cover environments where fsnotify is noisy or unavailable (huge workspaces, network filesystems).
  • (B) discards the existing debounce/ignore machinery and burns scans on idle workspaces; (C) alone fails the already-decided "live watch events" requirement.
  • The lock question is the real risk either way: refreshes and dashboard actions must serialize through withAppLock (non-reentrant), so the spec will require a single action/refresh queue regardless of which option you pick.
✅ DECISION (Jacob): (D) hybrid — with a sharpened reuse seam. The reuse point is RefreshWorkspaceForWatch(syncMode) (watch.go:48), not WatchWorkspace: the latter owns its own loop and writes to an io.Writer, so it is not drop-in embeddable in the tea loop. Instead a small tea.Cmd owns an fsnotify.Watcher, reuses the existing addWorkspaceWatches/watchEventRelevant filters, and emits WatchRefresh as a tea.Msg. The r key calls RefreshWorkspaceForWatch directly (useful even when fsnotify is live); --no-watch skips spawning the watcher cmd. Polling deliberately dropped — r covers the manual case without a ticker. As the context note flags, all refresh + action Cmds must serialize through a single queue because withAppLock is non-reentrant.

3. Test Strategy for the Dashboard

Spec 04's planning audit explicitly deferred the teatest/v2 adoption decision to this spec, where it becomes load-bearing. How should the dashboard be tested?

Current best-practice context: teatest/v2 is still under the experimental x/exp path (maintained — imports moved to charm.land in Mar 2026 — but never graduated), and its go.mod pins a bubbletea RC, so version friction is possible. However, a full-screen dashboard has far more interactive surface than spec 04's spinner/confirm, and the spec-04 audit's accepted risk ("manual evidence for TTY paths") was premised on that surface being tiny.
Recommended answer(s): (C)

Why these are recommended:

  • A dashboard's core logic is its message-driven state machine — (C) tests that both at the unit level (cheap, stable) and through real interaction flows (teatest), leaving vhs for what only a human needs to see.
  • (B) repeats spec 04's accepted gap at 10x the surface area; the planning audit will likely FLAG it again, and this time the risk is real.
  • (A) alone couples every assertion to rendered frames, which are brittle against theme/layout tweaks; keeping pure state-transition tests teatest-free limits that blast radius.
  • If teatest's RC pin causes dependency friction during implementation, the spec's escape hatch will be to drop to (B) for the affected flows and record the friction as a known limitation — you can pre-approve that fallback here.
⚠️ DECISION (Jacob): (B) — overrules the recommended (C). Do not adopt teatest.

Grounded in go.mod: the app is on the charm.land/*/v2 line (bubbletea/v2 v2.0.8, bubbles/v2, lipgloss/v2, huh/v2, fang/v2), and teatest/v2 is still experimental (x/exp, never graduated) and pins a bubbletea RC — a live version-friction risk against our pinned v2 line, in exchange for brittle golden-frame assertions the recommendation itself concedes are fragile.

The dashboard's core logic is its message-driven state machine, and that is fully reachable without a framework: thread message sequences through Update() in plain Go tests to cover both pure state transitions and interaction flows (navigate → trigger scan → assert resulting model state) — more stable than golden frames. The action Cmds (scan/hydrate/plan/apply-safe) are thin wrappers over already-tested domain functions, so they need no new harness. vhs proves the TTY render for humans. This is exactly the fallback (C) already names — we land on it directly, eyes open. Revisit teatest only if/when a v2-native harness graduates.
