Spec

05-spec-tui-dashboard

Sequence 05 · Feature: tui-dashboard · Phase 1

Introduction/Overview

DevSpace today exposes workspace state through a handful of separate commands (scan, status, doctor, plan, apply, project hydrate). A developer returning to a machine has to run several of them in sequence to answer a simple question: "what is the state of my workspace, and what safe thing should I do next?"

This feature adds devspace ui: a full-screen, read-first terminal dashboard (built on Bubble Tea v2) that renders the workspace in one place — a project table (hydration / dirty / env state), a workspace status pane, and a live watch-events pane — and lets the user trigger only the safe, non-destructive operations (scan, hydrate, plan, apply-safe) without leaving the interface. The primary goal is a single at-a-glance surface that reuses the existing domain functions rather than adding new logic, while preserving DevSpace's script-safe defaults and non-destructive-by-default invariants.

Goals

  • Provide a single interactive screen (devspace ui) that renders current workspace state — project table plus summary status — without running multiple commands.
  • Reuse the existing domain functions (ScanWorkspace, BuildPlan, ApplyLastPlan, HydrateProject) and the fsnotify watcher surface; add no new scan/plan logic and duplicate no existing machinery.
  • Serialize every refresh and action through a single withAppLock-guarded, single-flight queue so the non-reentrant app lock is never contended and the Bubble Tea update loop never blocks.
  • Keep the root command script-safe: the dashboard lives behind devspace ui; bare devspace behavior (help + non-interactive exit) is unchanged.
  • Ship automated tests on the model's state machine (plain Go tests on Update) plus a vhs recording as TTY proof — with no teatest dependency.

User Stories

  • As a developer returning to a machine, I want to open one screen showing which projects are hydrated, placeholder, missing, dirty, or lack an env file, so that I understand the workspace state without running scan, status, and doctor separately.
  • As a developer, I want to run scan, plan, apply-safe, and hydrate the selected project from within the dashboard, so that I can bring the workspace into alignment without memorizing or leaving for individual subcommands.
  • As a developer editing files in the workspace, I want the dashboard to reflect filesystem changes live, with a manual-refresh key and a --no-watch fallback, so that the view stays accurate even where fsnotify is noisy or unavailable.
  • As a script author, I want bare devspace to keep printing help and exiting non-interactively, so that existing automation is unaffected by the new interactive surface.

Demoable Units of Work

Three end-to-end vertical slices, in build order. Each slice is independently demoable and leaves the previous slices working. The build order is deliberate: render the state first, then introduce the serialized action queue (the load-bearing concurrency primitive), then feed live refreshes through that same queue.

Unit 1: devspace ui command + read-only dashboard

Purpose: Establish the command and a full-screen dashboard that renders workspace state from a real scan and quits cleanly. Smallest complete slice: launch → see state → quit. No actions, no live watch yet.

Functional Requirements:

  • The system shall register a ui subcommand on the root command via the existing NewRootCommand AddCommand pattern (commands.go), leaving cmd/devspace/main.go (the fang.Execute entrypoint) and bare-devspace behavior unchanged.
  • The system shall launch a full-screen (alt-screen) Bubble Tea v2 program implementing a single model with Init / Update(msg) (model, cmd) / View.
  • On startup the system shall perform one workspace scan and render a project table with, per project: name, type (git/local/external), status (Hydrated / Placeholder / Missing derived from ProjectState.Hydrated, .Placeholder, .Missing/.Exists), dirty flag (ProjectState.Dirty), current branch (ProjectState.CurrentBranch), and env presence (ProjectState.EnvFilePresent).
  • The system shall render a workspace status pane summarizing ScanSummary counts (found projects, git repos, untracked folders, local-only, projects-with-env) and the workspace root + machine identity in a header.
  • The user shall be able to navigate the project table (arrow keys and j/k) and quit at any time with q or ctrl-c.
  • Status shall be signaled with text labels (not color alone), using the existing styles.go theme and badge rendering for pass/warn/fail coloring.

Proof Artifacts:

  • CLI: devspace ui --help output demonstrates the command exists and documents its flags.
  • Test: a plain Go unit test (DEVSPACE_HOME temp workspace) asserts the initial model, after the startup scan message, holds the expected project rows and summary counts — demonstrates the render is driven by real scan data.
  • vhs: a recording of launch → navigate → quit under docs/specs/05-spec-tui-dashboard/05-proofs/ demonstrates the dashboard renders and exits cleanly on a TTY.

Unit 2: Safe actions through a serialized withAppLock queue

Purpose: Let the user trigger the safe operations from the dashboard, executed through a single-flight queue that holds the app lock for the whole operation and reports results back into the UI. This slice introduces the concurrency primitive the rest of the feature depends on.

Functional Requirements:

  • The user shall be able to trigger, from the dashboard: scan (ScanWorkspace), plan (BuildPlan), apply-safe (ApplyLastPlan), and hydrate the selected project (HydrateProject), each bound to a documented key shown in the footer.
  • The system shall execute each action inside a single serialized queue that acquires withAppLock for the full span of the operation and runs it in a tea.Cmd goroutine, posting the result back as a tea.Msg; the Update loop shall never block on an action.
  • The system shall be single-flight: while an action or refresh is in progress it shall not start a second one (queue or reject with a visible "busy" indicator), so that no two withAppLock acquisitions ever overlap.
  • The system shall reflect each action's lifecycle in the UI (pending → success/failure) and surface any returned error text without crashing the program.
  • The system shall expose only the four safe actions above; it shall not offer any destructive, sync, remote, hosted, or secret-editing operation.
  • Implementation shall verify against lock.go which reused functions already acquire withAppLock and must not double-wrap them (the app lock is non-reentrant; see Technical Considerations).

Proof Artifacts:

  • Test: a unit test drives an action message through Update and asserts the model transitions pending → result and that the underlying domain function's effect matches its CLI equivalent (temp DEVSPACE_HOME).
  • Test: a concurrency unit test asserts that a second action requested while one is in flight is queued/rejected rather than starting a second overlapping operation — demonstrates the single-flight lock discipline.
  • vhs: a "tour" recording (navigate → hydrate a placeholder → apply-safe) under 05-proofs/ demonstrates safe actions end-to-end.

Unit 3: Live watch pane + manual refresh + --no-watch

Purpose: Keep the view current as the filesystem changes, reusing the existing fsnotify pipeline and feeding refreshes through the Unit 2 queue, with a manual refresh key and an escape hatch for environments where fsnotify is unreliable.

Functional Requirements:

  • The system shall run a tea.Cmd that owns an fsnotify.Watcher, reuses the existing addWorkspaceWatches and watchEventRelevant filters, and emits refresh results as a tea.Msg carrying a WatchRefresh (from RefreshWorkspaceForWatch).
  • The system shall route watch-triggered refreshes through the same single-flight queue as actions (Unit 2), so refreshes and actions never contend for withAppLock.
  • The user shall be able to press r to force an immediate refresh, which works even when the live watcher is active and when --no-watch is set.
  • The system shall accept a --no-watch flag that skips spawning the watcher tea.Cmd entirely; the dashboard shall remain fully usable with manual r refresh only.
  • The system shall render a live events pane showing recent refresh events (e.g. changed-project / count / timestamp from WatchRefresh), most-recent first, bounded to a fixed number of lines.

Proof Artifacts:

  • Test: a unit test sends a WatchRefresh message (and, separately, an r-key message) through Update and asserts the model's table/summary/events pane update accordingly.
  • Test: a unit test with --no-watch asserts no watcher command is scheduled and that r still refreshes.
  • vhs: a recording (edit a tracked path in a temp workspace → dashboard reflects the change; then toggle --no-watch + r) under 05-proofs/ demonstrates live and manual refresh.

Non-Goals (Out of Scope)

To manage scope and protect DevSpace's safety posture, this feature will NOT include:

  1. Any destructive operation: no delete, overwrite, force, prune, or git reset/clean from the dashboard. Only the safe actions (scan, hydrate, plan, apply-safe) are exposed, preserving the non-destructive-by-default invariant and Safety tagging.
  2. Sync and remote management: no workspace push/pull, remote set/create, hosted control-plane config/serve, env-secret editing, or team/access/invite management from the UI — those stay CLI-only for this spec.
  3. Root-command interactivity: bare devspace stays non-interactive (help + exit); the dashboard is only reachable via devspace ui.
  4. teatest / golden-frame tests: no adoption of teatest/v2 and no rendered-frame golden files (see Technical Considerations for rationale).
  5. Manifest reconciliation / conflict view: deferred to spec 06 (manifest-reconciliation); the dashboard does not resolve conflicts.
  6. Mouse support and theme configuration: keyboard-only for this spec; reuse the existing lipgloss theme without adding user-configurable themes.

Design Considerations

Full-screen alt-screen Bubble Tea v2 layout, top to bottom:

  • Header: workspace root path + machine identity + sync mode.
  • Project table (main pane): columns Project · Type · Status · Dirty · Branch · Env. Status and Dirty use text labels plus theme color (pass/warn/fail badges from styles.go), never color alone.
  • Status/summary pane: ScanSummary counts.
  • Live events pane: recent watch refresh events, bounded line count (Unit 3).
  • Footer key-hint bar: shows active keybindings (navigate, refresh, the four actions, quit).

Accessibility: q/ctrl-c always quit; no color-only signaling; follow the existing accessible-mode conventions already used with huh in interactive.go. Rendering should degrade gracefully in narrow terminals (truncate columns rather than break layout).

Repository Standards

  • Architecture: single package internal/devspace; wire the command through NewRootCommand's AddCommand pattern in commands.go; keep cmd/devspace/main.go thin (fang.Execute wraps the root).
  • Reuse over reinvention: call existing domain functions and watch helpers; do not duplicate scan/plan logic.
  • Path safety: any workspace-relative path from the manifest or user input goes through safeWorkspacePath; never join workspace paths by hand.
  • Lock discipline: acquire withAppLock only at the outermost action/refresh boundary (the queue), never nested inside domain functions — matches the documented rule in lock.go.
  • Styling: reuse the styles.go theme and lipgloss/v2; prefer the lipgloss/v2 table renderer already used in output.go for render consistency (bubbles/v2 table is an acceptable alternative — see Open Questions).
  • Testing: tests set DEVSPACE_HOME via t.Setenv to a temp dir and operate on a temp workspace (per CLAUDE.md); use the commands_test.go executeCommand-style harness for command wiring. Gate with make verify (go test ./... + vet + lint + build).
  • Conventions: conventional commit messages; idempotent, non-destructive operations only.

Technical Considerations

Framework: Bubble Tea v2 (charm.land/bubbletea/v2), full-screen program with alt-screen. This is the repo's first full interactive model — only a minimal spinnerModel exists today in interactive.go. Styling via lipgloss/v2 + existing styles.go.

Watch integration: reuse point is RefreshWorkspaceForWatch(syncMode) plus the addWorkspaceWatches / watchEventRelevant filters — not WatchWorkspace, which owns its own loop and writes to an io.Writer and is therefore not embeddable in the tea message loop. A tea.Cmd owns the fsnotify.Watcher and emits WatchRefresh as a tea.Msg. Polling is deliberately excluded — the r key covers the manual case without a ticker.

Concurrency & the app lock (critical correctness constraint): withAppLock (lock.go) is non-reentrant and uses a bounded TryLockContext (~10s) — a nested or overlapping acquisition will fight the timeout. The reused domain functions (ScanWorkspace, BuildPlan, ApplyLastPlan, HydrateProject) and RefreshWorkspaceForWatch do not acquire the lock themselves; the CLI commands acquire it at their RunE boundary, and the watch loop acquires it in its runRefresh closure. Therefore the dashboard must: (1) run every action and every refresh through a single serialized queue that holds withAppLock for the whole operation; (2) be single-flight so two acquisitions never overlap; (3) execute the locked operation in a tea.Cmd goroutine and post the result back as a message so the Update loop never blocks. Implementation MUST re-read lock.go and the current call sites during Phase 3 to confirm no reused function is double-wrapped.

Test strategy (decided — no teatest): the app is on the charm.land/*/v2 line; teatest/v2 is experimental (x/exp, never graduated) and pins a bubbletea RC, a version-friction risk in exchange for brittle golden-frame assertions. Instead, thread message sequences through Update() in plain Go tests to cover both pure state transitions and interaction flows (navigate → trigger action → assert resulting model state) — more stable than golden frames. The action tea.Cmds are thin wrappers over already-tested domain functions, so they need no new harness. vhs provides the TTY render proof for humans.

Security Considerations

  • No new credentials or tokens are introduced. The dashboard triggers only existing safe operations; HydrateProject uses the existing git authentication path.
  • Env profiles are shown only as a boolean presence indicator (ProjectState.EnvFilePresent); the UI shall never render env contents, secret values, or decrypted profile data. Generated .env files remain 0600 and are not created by the dashboard.
  • Remote URLs may embed credentials; the table shows project/branch/status, and any remote display must not surface embedded tokens.
  • Proof artifact security: vhs recordings and screenshots must be captured against a sanitized temp workspace and must not commit real remote URLs with tokens, secret values, or private paths. Recordings live under 05-proofs/.

Success Metrics

  1. Launches and renders: devspace ui opens full-screen, renders the project table + status pane from a real scan, and quits cleanly on q/ctrl-c in 100% of manual runs.
  2. Script safety preserved: bare devspace behavior is unchanged — existing command tests continue to pass.
  3. Action parity: each dashboard safe action produces the same result as its CLI equivalent, verified by unit tests asserting model state after action messages.
  4. Zero lock contention: no two overlapping withAppLock acquisitions under normal use; enforced by the single-flight queue and covered by a concurrency unit test.
  5. Gate green: make verify passes; new state-transition tests added; at least one vhs recording committed as TTY proof.

Open Questions

Non-blocking; can be resolved during implementation without changing scope, requirements, demoable units, or proof artifacts:

  1. Table component: reuse the lipgloss/v2 table renderer already imported in output.go, or adopt the bubbles/v2 table component. Both satisfy the render requirement; prefer the lipgloss renderer for consistency unless bubbles/v2 table materially eases in-table selection/scrolling.
  2. Exact non-essential keybindings (beyond q/ctrl-c quit, r refresh, arrows/j/k navigate, and the four action keys) and the events-pane line budget may be tuned during implementation.
