Task 02 Proofs - Serialized withAppLock action queue + safe actions

Sequence 05 · Feature: tui-dashboard · Task 02 · Phase 3

Task Summary

Added the four safe actions (scan, plan, apply-safe, hydrate-selected) to the dashboard, each executed through a single lock boundary (runLocked → withAppLock) inside a tea.Cmd goroutine that posts its result back as a message. The model's busy flag enforces single-flight: while an operation is in progress no new locked command is dispatched, so two withAppLock acquisitions never overlap. The app lock is non-reentrant, which is documented at the runLocked boundary.

What This Task Proves

  • An action transitions the model pending→result, and apply-safe from the dashboard produces the same placeholder that ApplyLastPlan produces directly (action parity).
  • Single-flight holds: a second action requested while busy dispatches no command and shows a "busy" hint.
  • A failing action surfaces its error text in the model without panicking or quitting.

Evidence Summary

  • 3 tests pass: action transition + apply parity, single-flight rejection, action error handling.
  • The lock boundary is a single non-reentrant wrapper (runLocked) with an explanatory comment.

Artifact: action-queue tests

What it proves: action lifecycle, apply parity, single-flight lock discipline, and error surfacing all behave as specified.

Why it matters: the single-flight withAppLock queue is the load-bearing correctness constraint of the whole feature — overlapping acquisitions would fight the lock's bounded timeout. TestDashboardSingleFlightRejectsConcurrent is the guard against that regression.

Go test: 3 passed in 1 packages

Artifact: the single non-reentrant lock boundary

What it proves: every action/refresh goes through exactly one withAppLock wrapper; reused domain functions are not double-wrapped.

Why it matters: withAppLock is non-reentrant (per lock.go); a nested acquisition would deadlock against the lock's timeout. Codifying the boundary + comment prevents future callers from wrapping twice.

Source (internal/devspace/ui_actions.go):

Result summary: the queue is one helper with the non-reentrancy contract stated inline.

// runLocked is the only dashboard lock boundary. withAppLock is non-reentrant:
// domain functions called here must not acquire it themselves.
func runLocked(op func() error) error {
	return withAppLock(op)
}
