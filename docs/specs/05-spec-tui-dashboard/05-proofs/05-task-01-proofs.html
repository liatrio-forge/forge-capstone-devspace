Task 01 Proofs - devspace ui command + read-only dashboard

Sequence 05 · Feature: tui-dashboard · Task 01 · Phase 3

Task Summary

Added the devspace ui subcommand and a full-screen Bubble Tea v2 dashboard that renders workspace state from one scan (project table + summary pane + header + footer) and navigates/quits with the keyboard. The command refuses to launch on a non-TTY, keeping the CLI script-safe, and bare devspace behavior is unchanged.

What This Task Proves

  • The ui command exists, is registered on the root, and documents its --no-watch flag.
  • Launching without an interactive terminal fails fast and clearly (exit code 1), so scripts and pipes are unaffected.
  • The dashboard model renders real scan-derived rows and summary counts, and keyboard navigation/quit behave correctly — verified by plain-Go tests plus a headless program boot.

Evidence Summary

  • devspace ui --help lists the command and --no-watch.
  • devspace ui on a non-TTY exits 1 with "requires an interactive terminal".
  • Bare devspace still prints help (script-safe posture preserved).
  • 3 unit/lifecycle tests pass (render, navigate/quit, headless render smoke).

Artifact: ui --help output

What it proves: the command is wired into the root command tree with its flag documented.

Why it matters: discoverability is the whole point of the dashboard; the help output is the first contract a user sees.

Command:

bin/devspace ui --help

Result summary: help renders with the description and both --no-watch and the inherited --no-color flags.

  Open a full-screen dashboard showing tracked projects, workspace scan counts,
  and recent filesystem refreshes.

  The dashboard exposes only safe actions: scan, plan, apply-safe, and hydrate
  the selected project.

  USAGE
    devspace ui [--flags]

  FLAGS
    -h --help   Help for ui
    --no-color  Disable styled output regardless of terminal capability
    --no-watch  Disable the live filesystem watcher; use the r key to refresh manually

Artifact: non-TTY guard

What it proves: the dashboard refuses to run when stdout is not an interactive terminal, exiting non-zero with a clear message.

Why it matters: DevSpace's posture is script-safe by default; an interactive full-screen app must never hang or emit control codes into a pipe/CI log.

Command:

bin/devspace ui < /dev/null > out.txt 2>&1 ; echo "rc=$?"

Result summary: exit code 1 with a styled ERROR block reading "Devspace ui requires an interactive terminal."

   ERROR
  Devspace ui requires an interactive terminal.
rc=1

Artifact: bare devspace unchanged

What it proves: adding the subcommand did not change the root command's non-interactive help/exit behavior.

Why it matters: spec non-goal — bare devspace must stay script-safe; existing automation depends on it.

Command:

bin/devspace

Result summary: prints the standard root help ("Synchronize local developer workspace metadata …"), exactly as before.

Artifact: unit + lifecycle tests

What it proves: the model renders real scan data, navigation clamps correctly, quit works, and the program boots+exits headlessly.

Why it matters: these are the render/state-machine contracts; the headless smoke test (audit remediation) covers the real render + program lifecycle without teatest.
