Overview

Four parent tasks. Tasks 1.0–3.0 map one-to-one to the spec's three demoable units in build order (render → serialized action queue → live watch). Task 4.0 collects the cross-cutting proof artifacts (vhs), documentation, and the full make verify gate. Implementation routes to codex exec; sub-tasks are written test-first where practical, per the repo's Go testing conventions.

Notes

  • Put tests in internal/devspace/ui_test.go; name tests after behavior (e.g. TestDashboardSingleFlightRejectsConcurrent), per AGENTS.md.
  • Every test that reads/writes config/state/manifest must set t.Setenv("DEVSPACE_HOME", tmp) and operate on a temp workspace — never touch the real ~/.devspace.
  • Run the gate with make verify (test + vet + lint + vulncheck + build). Lint enforces gofmt -l cmd internal empty and golangci-lint incl. gosec.
  • Do not commit real remotes, tokens, age identities, or generated state into proof artifacts; use placeholders / temp workspaces (AGENTS.md security rule).
  • Test strategy is decided: plain-Go Update tests + vhs; no teatest (see spec Technical Considerations).

1.0 devspace ui command + read-only dashboard (Unit 1)

1.0 Proof Artifact(s)

  • CLI: go run ./cmd/devspace ui --help output demonstrates the command and its --no-watch flag exist and are documented.
  • CLI: go run ./cmd/devspace ui < /dev/null (not a TTY) exits non-zero with a clear "requires an interactive terminal" error — demonstrates the script-safe TTY guard.
  • Test: go test ./internal/devspace -run TestDashboardInitialModelRendersScan -v passes — initial model after the scan message holds expected rows (status/dirty/branch/env) and ScanSummary counts on a temp workspace.
  • Test: go test ./internal/devspace -run TestDashboardNavigateAndQuit -v passes — arrow/j/k move the selection and q/ctrl-c return tea.Quit.
  • Test: go test ./internal/devspace -run TestDashboardHeadlessRenderSmoke -v passes — boots the Bubble Tea program with in-memory input/output (no teatest), seeds state, and asserts the rendered output contains the header/table and the program exits cleanly. (audit remediation: covers the real render/lifecycle path)

1.0 Tasks

  • 1.1 Create ui.go with newUICommand() *cobra.Command (a --no-watch bool flag) and register it in NewRootCommand via cmd.AddCommand(newUICommand()). No lock acquired at construction.
  • 1.2 In the command's RunE, add a TTY guard: if stdout is not a terminal, return a clear error and non-zero exit; otherwise launch the Bubble Tea program with alt-screen. Reuse the repo's existing terminal-detection path if one exists (it drives styled output).
  • 1.3 In ui_model.go, define Model (project rows derived from ProjectState, ScanSummary counts, selected index, events slice, busy flag, errText, noWatch bool) and message types (scanLoadedMsg, actionResultMsg, watchRefreshMsg, errMsg).
  • 1.4 Implement Init (fire the initial scan/refresh command) and the Update/View for the read-only path: populate rows+summary on scanLoadedMsg, handle window-size, and render header (workspace root + machine id + sync mode), the project table (Project · Type · Status · Dirty · Branch · Env with text labels + theme badges), the summary pane, and the footer key-hint bar. Reuse styles.go + lipgloss/v2 table renderer.
  • 1.5 Implement navigation (arrows + j/k, clamped to row bounds) and quit (q/ctrl-c → tea.Quit). Status must never be color-only; degrade gracefully in narrow widths.
  • 1.6 Write TestDashboardInitialModelRendersScan and TestDashboardNavigateAndQuit in ui_test.go (temp DEVSPACE_HOME, seeded workspace; thread scan-loaded + key messages through Update; assert rows/summary/selection/quit).
  • 1.7 Capture the ui --help output and the non-TTY guard behavior for the proof artifacts.
  • 1.8 (audit remediation) Add TestDashboardHeadlessRenderSmoke in ui_test.go: construct the program with tea.WithInput/tea.WithOutput over in-memory buffers, send a scan-loaded message then a quit key, and assert Run returns nil and the output buffer contains the header + a table row. Closes the "real render only in vhs" gap without teatest.

2.0 Serialized withAppLock action queue + safe actions (Unit 2)

2.0 Proof Artifact(s)

  • Test: go test ./internal/devspace -run TestDashboardActionTransitions -v passes — an action message drives pending→result and the effect (e.g. apply-safe placeholder creation) matches the CLI equivalent on a temp workspace.
  • Test: go test ./internal/devspace -run TestDashboardSingleFlightRejectsConcurrent -v passes — a second action while one is in flight is queued/rejected; no two withAppLock acquisitions overlap.
  • Test: go test ./internal/devspace -run TestDashboardActionError -v passes — a failing action surfaces its error text without panicking or quitting.
  • CLI: workspace files before/after an apply-safe triggered in the dashboard equal the result of devspace apply — demonstrates action parity and no destructive changes.

2.0 Tasks

  • 2.1 Re-read lock.go and confirm which reused functions acquire withAppLock (expected: none of ScanWorkspace/BuildPlan/ApplyLastPlan/HydrateProject/RefreshWorkspaceForWatch self-lock). Record the finding in a comment at the queue so the non-reentrancy contract is explicit.
  • 2.2 In ui_actions.go, implement the single-flight queue: a helper that wraps an operation func() (payload, error) in withAppLock, runs it inside a tea.Cmd goroutine, and returns actionResultMsg/errMsg. The Update loop must never block on it.
  • 2.3 Add tea.Cmd constructors for the four safe actions — scan (ScanWorkspace), plan (BuildPlan), apply-safe (ApplyLastPlan), hydrate-selected (HydrateProject on the selected row) — each routed through the queue, bound to a documented key, and shown in the footer. Any path handling goes through safeWorkspacePath. Expose no destructive/sync/hosted/secret action.
  • 2.4 Update Update: set busy=true on action start and reject a new action while busy; on actionResultMsg/errMsg clear busy, apply the result (re-render affected rows/summary) or show the error text in the status line without quitting.
  • 2.5 Write TestDashboardActionTransitions, TestDashboardSingleFlightRejectsConcurrent, and TestDashboardActionError in ui_test.go.
  • 2.6 Manual parity check: trigger apply-safe from the dashboard vs devspace apply on the same temp workspace; confirm identical placeholder creation and no destructive changes. Capture for the proof artifact.
