Task 04 Proofs - Expanded --json scripting interface

Sequence 04 · Feature: charm-terminal-output · Task 04 · Phase 3

Task Summary

This task added --json to status, doctor, workspace diff, and mount --preview, joining the existing plan --json and setup plan --json. Rather than inventing parallel report types, existing structs were reused and given stable JSON tags where they didn't already have them (ManifestDiff/ProjectDiff/FieldChange, the internal doctorReport/doctorCheck), and one new struct (WorkspaceStatusReport) was introduced for the aggregate status view, with a single buildWorkspaceStatusReport function now the shared source of truth for both the text and JSON rendering paths — eliminating what had been duplicated counting logic.

What This Task Proves

  • All four new --json outputs are valid, parseable JSON with the exact field names asserted by four new schema-stability tests.
  • No new report structs were invented where an existing one already fit: mount --preview --json reuses MountEntry (already had JSON tags), workspace diff --json reuses ManifestDiff (tags added), doctor --json reuses the internal doctorReport (tags added).
  • The status command's aggregate counting logic is no longer duplicated between the text and JSON code paths — both call buildWorkspaceStatusReport().
  • The README's "Supported Commands" section documents all four new flags plus the --no-color flag and the piped-output guarantee.

Evidence Summary

  • CLI transcripts: all four --json outputs piped through python3 -m json.tool, confirming valid JSON with real data from a sandboxed workspace.
  • Test suite: four new schema-stability tests plus the full existing suite — 139 test functions pass, 0 failed.
  • Quality gates: go vet, golangci-lint v2.12.2, and govulncheck all clean.

Artifact: status --json parses and reports the same counts as the text view

What it proves: WorkspaceStatusReport serializes with stable camelCase field names and the same underlying counts the styled text view shows.

Why it matters: Scripts can now poll workspace health (tracked/hydrated/dirty/missing-env counts) without scraping colored text.

Artifact: workspace diff --json reuses ManifestDiff with real added/removed/changed data

Why it matters: No duplicate diff type was needed — adding JSON tags to the existing struct was sufficient, keeping one source of truth for the diff shape.

Reviewer Conclusion

The scripting interface now covers every major read command. Reusing existing structs (adding tags rather than inventing parallel types) kept the diff small and avoided a second source of truth for diff/mount/doctor shapes. The status command's refactor is a net simplification — one function now serves both the text and JSON views instead of two copies of the same counting loop.
