Sequence 06 · Feature: manifest-reconciliation · Phase 2 · Sub-tasks generated

Overview

Four parent tasks. Tasks 1.0–3.0 map to the spec's three demoable units in build order (track the base → merge engine + git reconcile → hosted reconcile, reusing the engine per Q3=C). Task 4.0 is the cross-cutting demo, docs, and make verify gate. Implementation routes to codex exec; the pure merge engine (Task 2.0) gets the tightest spec and the heaviest test coverage.

Codebase finding (drives Task 2.0): a three-way, conflict-reporting merge engine already exists at internal/devspace/manifest_merge.go — mergeManifests(base, ours, theirs) (Manifest, []MergeConflict, error), keyed by project ID with per-record and access-grant merging, covered by manifest_merge_test.go — but it has no production callers. Task 2.0 wires and extends this engine (two-way fallback, ops summary) instead of writing a new one.

Notes

  • Tests live beside implementation in internal/devspace/, isolate state with t.Setenv("DEVSPACE_HOME", t.TempDir()) plus a temp workspace, and are behavior-named (e.g. TestWorkspaceReconcileConflictBlocksApply).
  • Local gate: make verify = test + vet + golangci-lint (incl. gosec) + gofmt -l cmd internal empty + govulncheck + build. Run per parent task and before handoff.
  • Invariants: never hand-join workspace paths (use safeWorkspacePath / SaveManifest); withAppLock is non-reentrant — acquire only at the command boundary; all new JSON writes go through the atomic jsonio helpers with 0o600 perms.
  • Non-destructive posture: reconcile never writes the manifest while unresolved conflicts exist; apply always backs up first and is hash-guarded; plain pull/push behavior is unchanged.
  • Conventional Commit prefixes (feat:, docs:…); one scoped change per commit.
  • Open-question resolutions baked into these sub-tasks: global --force-local/--force-remote (per-project selection deferred); hosted surface is a dedicated devspace hosted reconcile subcommand; a single base snapshot (matches the single-remote model).

Tasks

1.0 Track the last-synced base manifest (Unit 1)

Record the synced manifest as a base snapshot in app home after every successful pull/push (git and hosted), plus a loader with a clear "no base" signal. This is the enabling primitive for real three-way conflict detection (Q2=A).

1.0 Proof Artifact(s)

  • Test: go test ./internal/devspace -run TestSyncRecordsBaseManifest -v passes — a successful git workspace pull and push each write a base snapshot equal to the synced manifest (temp DEVSPACE_HOME).
  • Test: go test ./internal/devspace -run TestHostedSyncRecordsBaseManifest -v passes — a successful hosted pull/push writes the base snapshot too.
  • Test: go test ./internal/devspace -run TestBaseManifestAbsentIsDetectable -v passes — the loader reports "no base recorded" cleanly when none exists.
  • Artifact: the base file (last-synced-manifest.json) is written atomically with app-home state permissions and contains manifest metadata only (no secrets/machine-local paths).

1.0 Tasks

  • 1.1 Add baseManifestPath() to paths.go, resolving to <app home>/last-synced-manifest.json via the same appHome() resolution as configPath()/statePath() (honors DEVSPACE_HOME/DEV_DROP_HOME).
  • 1.2 Create base_manifest.go with recordBaseManifest(m Manifest) error (atomic writeJSON, 0o600) and loadBaseManifest() (Manifest, bool, error) that returns ok=false (no error) when the snapshot is absent, using the existing missing(err) helper.
  • 1.3 Call recordBaseManifest on the success paths of PullWorkspaceManifest and PushWorkspaceManifest in workspace_sync.go, after the sync itself has fully succeeded, recording exactly the manifest content that was synced.
  • 1.4 Call recordBaseManifest on the success paths of PullHostedManifest and PushHostedManifest in hosted_sync.go, alongside the existing recordHostedSync(version, hash) bookkeeping.
  • 1.5 Write TestSyncRecordsBaseManifest, TestHostedSyncRecordsBaseManifest (in-process hosted server harness), and TestBaseManifestAbsentIsDetectable — each with t.Setenv("DEVSPACE_HOME", ...) + temp workspace, asserting snapshot content equals the synced manifest and file perms are 0o600.

2.0 Three-way merge engine + workspace reconcile (git) (Unit 2)

The backend-agnostic Reconcile(base, local, remote) function (Q1=D hybrid) and the review-first git command (Q4=A): localize remote, merge against the base, write a reviewable reconcile-plan, back up + hash-guard the apply, and require explicit force flags to resolve genuine conflicts. Two-way fallback when no base exists.

2.0 Proof Artifact(s)

  • Test: go test ./internal/devspace -run TestReconcileMergeCases -v passes — table test: add/remove/change on one side auto-merge; change-both / add-add-different are conflicts; no-base two-way fallback flags all same-path diffs.
  • Test: go test ./internal/devspace -run TestWorkspaceReconcileNonConflicting -v passes — divergent-but-mergeable manifests reconcile to the union, back up the old manifest, and apply under the hash guard.
  • Test: go test ./internal/devspace -run TestWorkspaceReconcileConflictBlocksApply -v passes — a genuine conflict stops apply; --force-local/--force-remote resolves it explicitly.
  • CLI: devspace workspace reconcile --json output demonstrates auto-merged ops + the conflict list; --help documents the command and flags.

2.0 Tasks

  • 2.1 Create reconcile.go with ReconcileResult (merged manifest, added/removed/changed op summaries, []MergeConflict) and reconcileManifests(base *Manifest, local, remote Manifest) (ReconcileResult, error) that wraps the existing mergeManifests from manifest_merge.go; a nil base triggers the documented two-way fallback: every same-key difference between local and remote becomes a conflict (no auto-merge, never a silent overwrite), with additions/removals unique to one side still merging.
  • 2.2 Write table test TestReconcileMergeCases in reconcile_test.go: add-one-side, remove-one-side, change-one-side (auto-merge); change-both-different, add-add-different, change-vs-remove (conflict); no-base fallback flags all same-path diffs; re-reconciling an already-merged state yields zero ops (idempotency).
  • 2.3 Add a persisted ReconcilePlan (ops, conflicts with text labels, local-manifest hash at plan time, remote source info) written atomically to app home via a new reconcilePlanPath() — the review-first artifact mirroring last-plan.json — plus load/save helpers.
  • 2.4 Implement ReconcileWorkspaceManifest(force string, apply bool): localize the remote manifest by reusing the fetch machinery behind DiffWorkspaceManifest, load the base (1.2), run reconcileManifests, write the plan (2.3); on apply, re-verify the local manifest hash against the plan (mirroring apply's guard), back up the pre-reconcile manifest to app home, then write the merged manifest through SaveManifest and record the new base.
  • 2.5 Conflict semantics: when conflicts exist and no force flag is given, stop before any write and report each conflict with text labels (project key, field, local vs remote values); --force-local/--force-remote resolve all conflicts to that side globally (mutually exclusive flags; per-project selection is deferred by design).
  • 2.6 Add newWorkspaceReconcileCommand() in commands.go, registered under newWorkspaceCommand(), with --json, --apply, --force-local, --force-remote; wrap the run in withAppLock at the command boundary only; text output follows the diff/plan styling conventions in output.go/styles.go.
  • 2.7 Write TestWorkspaceReconcileNonConflicting (union result, backup file written, hash guard enforced, idempotent re-run) and TestWorkspaceReconcileConflictBlocksApply (conflict blocks apply; each force flag resolves it) against a temp workspace + local bare-repo remote.

3.0 Hosted-backend reconciliation (Unit 3)

Reuse the Task 2.0 engine on the hosted path (Q3=C): on an HTTP 409 version conflict, fetch the current hosted manifest + version/hash, three-way merge against the base, present the same review artifact, and on apply push with the updated baseline (then refresh the base snapshot).

3.0 Proof Artifact(s)

  • Test: go test ./internal/devspace -run TestHostedReconcileResolvesVersionConflict -v passes — a simulated 409 → reconcile → apply pushes the merged manifest with the correct updated version/hash baseline (in-process hosted server harness).
  • Test: go test ./internal/devspace -run TestHostedReconcileConflictBlocksPush -v passes — a genuine conflict blocks the hosted push until an explicit force choice is given.
  • Test: go test ./internal/devspace -run TestHostedReconcileUpdatesBaseOnSuccess -v passes — a successful hosted reconcile refreshes the base snapshot (Unit 1 integration).

3.0 Tasks

  • 3.1 Implement ReconcileHostedManifest(force string, apply bool) in reconcile.go: fetch the current hosted envelope via hostedClient.get (manifest + version + hash), validate it with the existing envelope checks, load the base, run the shared reconcileManifests, and write the same ReconcilePlan artifact (tagged with the hosted source + remote version).
  • 3.2 Apply path: push the merged manifest with hostedClient.put(ctx, fetchedRemoteVersion, merged) so the server's optimistic-concurrency check accepts it; on success, back up + write the merged manifest locally (same guard as 2.4), then recordHostedSync(newVersion, hash) and recordBaseManifest(merged).
  • 3.3 Add newHostedReconcileCommand() registered under newHostedCommand() (dedicated devspace hosted reconcile subcommand — resolves the open question in favor of the existing command-group layout), sharing flags (--json, --apply, --force-local, --force-remote), output rendering, and withAppLock discipline with 2.6.
