Spec

10-spec-project-listing

Sequence 10 · Feature: project-listing · Phase 1

Introduction/Overview

DevSpace already tracks projects in the workspace manifest, but the project command group does not have a direct list view. This feature makes bare devspace project show tracked projects, including useful local state, without mutating the workspace or contacting remotes.

Goals

  • Make devspace project print a readable table of tracked projects.
  • Keep existing subcommands such as add, hydrate, remove, and status unchanged.
  • Show operational fields in the default view: name, path, type, hydration status, dirty flag, branch, and env-file presence.
  • Add a stable --json mode that returns full manifest project objects plus current project state.
  • Keep the listing read-only; users run devspace scan when they want refreshed metadata.

User Stories

  • As a DevSpace user, I want to run devspace project so that I can quickly see which projects are tracked.
  • As a user moving between machines, I want to see hydration and missing-state information so that I can decide which projects need devspace project hydrate or a scan.
  • As a script author, I want to run devspace project --json so that automation can consume project manifest metadata and local state without reading private files directly.

Demoable Units of Work

These units focus on one read-only CLI flow, then its scripting contract and docs/tests.

Unit 1: Bare Project Listing

Purpose: Let users view tracked projects from the existing project command group.

Functional Requirements:

  • The system shall make bare devspace project execute a list view instead of only showing command help.
  • The system shall preserve all existing project subcommands and their argument validation.
  • The system shall print one row per manifest project, sorted by relative path.
  • The system shall print a clear empty-state message when no projects are tracked.

Proof Artifacts:

  • Test: command test covering devspace project with two tracked projects demonstrates the new bare command output.
  • CLI transcript: devspace project in a sandbox workspace demonstrates the table and empty state.

Unit 2: Operational Table Fields

Purpose: Show useful project health without requiring users to open the TUI or inspect manifest/state files.

Functional Requirements:

  • The system shall include name, path, type, hydration status, dirty flag, current branch, and env-file presence in the default table.
  • The system shall derive hydration status from saved state using the same meanings as the dashboard: hydrated, placeholder, or missing.
  • The system shall not run ScanWorkspace, push, pull, hydrate, apply plans, or modify manifest/state as part of listing.
  • The system shall keep output readable when branch or state fields are absent by using a stable placeholder such as -.

Proof Artifacts:

  • Test: formatter or command test with hydrated, placeholder, and missing states demonstrates table status values.
  • CLI transcript: listing output demonstrates the selected columns and read-only behavior after setup.

Unit 3: JSON Project Listing

Purpose: Give scripts a stable view of manifest projects plus current local state.

Functional Requirements:

  • The system shall add --json to the listing command.
  • The JSON response shall include an array of rows containing each full manifest Project object and its matching ProjectState.
  • The JSON response shall remain free of ANSI styling and human table decorations.
  • The JSON response shall not expose secrets or plaintext env values.

Proof Artifacts:

  • Test: json.Unmarshal on devspace project --json output demonstrates stable field names and valid JSON.
  • CLI transcript: devspace project --json | python3 -m json.tool demonstrates parseable manifest-plus-state rows.

Unit 4: Documentation And Discovery

Purpose: Make the new read command visible in help and README usage.

Functional Requirements:

  • The system shall document devspace project and devspace project --json in the README command reference.
  • The system shall keep Cobra help text accurate for the project command group and its subcommands.
  • The system shall clarify that listing uses saved metadata and devspace scan refreshes it.

Proof Artifacts:

  • Diff: README update demonstrates user-facing documentation for the new command behavior.
  • Test: help or command test demonstrates --json is accepted on the project listing command.

Non-Goals (Out of Scope)

This feature is a project listing surface only.

  1. No implicit refresh: devspace project will not scan the workspace before printing.
  2. No remote sync: listing will not push, pull, reconcile, or contact hosted sync.
  3. No hydration action: listing will not clone or create project directories.
  4. No interactive UI changes: the existing devspace ui dashboard is not changed by this spec.
  5. No project filtering or search: filters, sorting flags, and detail subcommands can be considered later if needed.

Design Considerations

The default output should follow existing terminal-output patterns: concise labels, stable columns, color only through existing style helpers, and clean fallback when output is piped or --no-color is used. The JSON mode should be treated as the scripting surface and should not include human-only formatting.

Repository Standards

  • DevSpace is a Go CLI with command wiring in internal/devspace/commands.go and presentation helpers in internal/devspace/output.go.
  • Use standard Go formatting and keep tests next to implementation as *_test.go.
  • Prefer explicit user-facing command errors over silent fallbacks.
  • Keep CLI behavior local-first unless a command explicitly opts into Git or hosted sync.
  • Use existing JSON helpers and command-test patterns from commands_test.go.
  • Run make verify before completion.

Technical Considerations

No latest external standards research was needed; the feature uses existing Go, Cobra, manifest/state, output, and JSON patterns already present in the repository. Implementation should reuse existing types where possible: Project and ProjectState for JSON, and existing dashboard/status state derivation for human-readable hydration labels. Avoid adding dependencies or a new abstraction unless the existing output helpers cannot represent the table cleanly.

Security Considerations

  • The command must not print plaintext env values or encrypted profile contents.
  • The command may print manifest metadata already stored in manifest.json, including Git remote URLs in JSON mode because the user selected full manifest project objects.
  • The command must not read or write hosted sync tokens, age identities, or generated workspace state outside normal manifest/state loading.
  • Proof artifacts should use sandbox workspaces and placeholders, not real .env files or private remotes.

Success Metrics

  1. Bare command works: devspace project lists tracked projects and exits successfully in a configured workspace.
  2. Stable JSON works: devspace project --json parses as JSON and includes full project and state data for each row.
  3. Read-only behavior is preserved: listing does not change manifest, state, project files, remotes, or hosted sync.
  4. Verification passes: focused command tests and make verify pass.
